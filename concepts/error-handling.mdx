Switchboard uses explicit, type-safe error handling inspired by Rust's `Result` type. This page explains the error handling strategy and best practices.

## The Result type pattern

Instead of throwing exceptions, Switchboard uses the `Result<T>` type for explicit error handling:

```typescript
// From packages/core/src/types/result.ts
export type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };
```

Every operation that can fail returns a `Result`:

```typescript
const result = await bot.sendMessage(channelId, 'Hello');
//    ^? Result<UnifiedMessage>

if (result.ok) {
  console.log('Message sent:', result.value.id);
  //                          ^? UnifiedMessage
} else {
  console.error('Failed:', result.error.message);
  //                       ^? Error
}
```

### Why Result instead of exceptions?

**Type safety**: Errors are part of the function signature:

```typescript
// You know this can fail by looking at the return type
sendMessage(id: string, text: string): Promise<Result<UnifiedMessage>>

// vs throwing exceptions (hidden failure mode)
sendMessage(id: string, text: string): Promise<UnifiedMessage>  // Can throw!
```

**Explicit handling**: You must check the result:

```typescript
// ✅ Forces you to handle errors
const result = await bot.sendMessage(channelId, text);
if (!result.ok) {
  // Handle error
}

// ❌ Exceptions can be forgotten
try {
  await bot.sendMessage(channelId, text);
} catch (error) {
  // Easy to forget this block
}
```

**No try/catch noise**: Cleaner code flow:

```typescript
// Result pattern
const result = await bot.sendMessage(channelId, 'Hello');
if (!result.ok) return;

const message = result.value;
// Continue with message...

// vs try/catch
try {
  const message = await bot.sendMessage(channelId, 'Hello');
  // Continue with message...
} catch (error) {
  // Handle error
}
```

<Tip>
The Result pattern is inspired by Rust, Haskell's Either type, and functional programming error handling.
</Tip>

## Working with Result

### Basic usage

```typescript
const result = await bot.sendMessage(channelId, 'Hello');

if (result.ok) {
  // Success case
  const message = result.value;
  console.log('Sent:', message.id);
} else {
  // Error case
  const error = result.error;
  console.error('Failed:', error.message);
}
```

### Early return pattern

```typescript
bot.onMessage(async (ctx) => {
  const result = await bot.sendMessage(channelId, 'Hello');
  
  // Early return on error
  if (!result.ok) {
    console.error('Failed to send:', result.error);
    return;
  }
  
  // Continue with success value
  const message = result.value;
  await bot.addReaction(message, 'thumbsup');
});
```

### Helper functions

Switchboard provides helper functions for creating Results:

```typescript
import { ok, err, wrapAsync, isOk, isErr } from '@aarekaz/switchboard-core';

// Create success result
const success = ok({ id: '123', text: 'Hello' });
//    ^? Result<{ id: string, text: string }>

// Create error result
const failure = err(new Error('Something went wrong'));
//    ^? Result<never, Error>

// Wrap async operations
const result = await wrapAsync(async () => {
  const response = await fetch('https://api.example.com');
  return response.json();
});

// Type guards
if (isOk(result)) {
  console.log(result.value);
}

if (isErr(result)) {
  console.error(result.error);
}
```

### Pattern matching style

```typescript
// Functional approach
const result = await bot.sendMessage(channelId, 'Hello');

const message = result.ok 
  ? result.value.id
  : null;

// Or with early return
if (!result.ok) {
  console.error(result.error);
  return;
}

const message = result.value;
```

## Error types

Switchboard defines a hierarchy of error classes:

```typescript
// From packages/core/src/utils/errors.ts

// Base error class
export class SwitchboardError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly platform?: PlatformType
  ) {
    super(message);
    this.name = 'SwitchboardError';
  }
}
```

### Error hierarchy

```
SwitchboardError (base class)
├── AdapterNotFoundError
├── ConnectionError
├── MessageSendError
├── MessageEditError
├── MessageDeleteError
└── ReactionError
```

### Specific error types

#### AdapterNotFoundError

```typescript
export class AdapterNotFoundError extends SwitchboardError {
  constructor(platform: PlatformType) {
    super(
      `No adapter found for platform: ${platform}. Did you import @aarekaz/switchboard-${platform}?`,
      'ADAPTER_NOT_FOUND',
      platform
    );
  }
}
```

Thrown when:
- You try to create a bot for a platform without importing its adapter

```typescript
import { createBot } from '@aarekaz/switchboard-core';
// Forgot to import adapter!

const bot = createBot({ platform: 'discord' });
// → AdapterNotFoundError: No adapter found for platform: discord
```

#### ConnectionError

```typescript
export class ConnectionError extends SwitchboardError {
  constructor(platform: PlatformType, cause: unknown) {
    super(
      `Failed to connect to ${platform}: ${cause instanceof Error ? cause.message : String(cause)}`,
      'CONNECTION_ERROR',
      platform
    );
    this.cause = cause;
  }
}
```

Thrown when:
- Authentication fails
- Network connection fails
- Platform API is unreachable

```typescript
await bot.start();
// → ConnectionError: Failed to connect to discord: Invalid token
```

#### MessageSendError

```typescript
export class MessageSendError extends SwitchboardError {
  constructor(
    platform: PlatformType,
    channelId: string,
    cause: unknown
  ) {
    super(
      `Failed to send message to channel ${channelId} on ${platform}: ${cause instanceof Error ? cause.message : String(cause)}`,
      'MESSAGE_SEND_ERROR',
      platform
    );
    this.cause = cause;
  }
}
```

Returned when:
- Channel doesn't exist
- Bot lacks permissions
- Message content is invalid
- Rate limit exceeded

```typescript
const result = await bot.sendMessage('invalid-channel', 'Hello');
if (!result.ok) {
  console.error(result.error);
  // → MessageSendError: Failed to send message to channel invalid-channel
}
```

#### MessageEditError / MessageDeleteError

```typescript
export class MessageEditError extends SwitchboardError {
  constructor(platform: PlatformType, messageId: string, cause: unknown)
}

export class MessageDeleteError extends SwitchboardError {
  constructor(platform: PlatformType, messageId: string, cause: unknown)
}
```

Returned when:
- Message doesn't exist
- Bot doesn't own the message (can't edit)
- Bot lacks delete permissions
- Message is too old (some platforms limit edit time)

#### ReactionError

```typescript
export class ReactionError extends SwitchboardError {
  constructor(
    platform: PlatformType,
    messageId: string,
    emoji: string,
    cause: unknown
  )
}
```

Returned when:
- Message doesn't exist
- Emoji is invalid
- Bot lacks reaction permissions

### Checking error types

```typescript
const result = await bot.sendMessage(channelId, 'Hello');

if (!result.ok) {
  // Check specific error type
  if (result.error instanceof MessageSendError) {
    console.error('Send failed:', result.error.message);
    console.log('Platform:', result.error.platform);
    console.log('Code:', result.error.code);
  }
  
  // Check by error code
  if (result.error.code === 'MESSAGE_SEND_ERROR') {
    // Handle send error
  }
}
```

<Note>
All error classes include a `code` field for programmatic error handling and an optional `platform` field.
</Note>

## Best practices

### 1. Always check results

```typescript
// ✅ Good
const result = await bot.sendMessage(channelId, text);
if (!result.ok) {
  console.error('Failed to send message:', result.error);
  return;
}

// ❌ Bad: Ignoring errors
const result = await bot.sendMessage(channelId, text);
const message = result.value;  // Runtime error if result.ok is false!
```

### 2. Use type narrowing

```typescript
const result = await bot.sendMessage(channelId, text);

if (result.ok) {
  // TypeScript knows result.value exists
  console.log(result.value.id);
  //               ^? UnifiedMessage
} else {
  // TypeScript knows result.error exists
  console.error(result.error.message);
  //                   ^? Error
}
```

### 3. Handle specific error types

```typescript
const result = await bot.sendMessage(channelId, text);

if (!result.ok) {
  if (result.error instanceof MessageSendError) {
    // Specific handling for send errors
    await notifyAdmin(`Failed to send to ${channelId}`);
  } else if (result.error instanceof ConnectionError) {
    // Handle connection issues
    await reconnect();
  } else {
    // Generic error handling
    console.error('Unknown error:', result.error);
  }
}
```

### 4. Propagate errors when appropriate

```typescript
async function sendWelcomeMessage(userId: string): Promise<Result<UnifiedMessage>> {
  const dmChannel = await bot.createDM(userId);
  if (!dmChannel.ok) {
    return dmChannel;  // Propagate error
  }
  
  return bot.sendMessage(dmChannel.value.id, 'Welcome!');
}
```

### 5. Log errors with context

```typescript
const result = await bot.sendMessage(channelId, text);

if (!result.ok) {
  console.error('Failed to send message', {
    error: result.error.message,
    code: result.error.code,
    platform: result.error.platform,
    channelId,
    text: text.substring(0, 50)  // Log preview
  });
}
```

### 6. Don't mix Result with exceptions

```typescript
// ✅ Good: Consistent error handling
bot.onMessage(async (ctx) => {
  const result = await bot.sendMessage(channelId, 'Hello');
  if (!result.ok) {
    console.error(result.error);
    return;
  }
  
  // Continue...
});

// ❌ Bad: Mixing styles
bot.onMessage(async (ctx) => {
  try {
    const result = await bot.sendMessage(channelId, 'Hello');
    if (!result.ok) throw result.error;  // Don't do this
  } catch (error) {
    // ...
  }
});
```

<Warning>
Event handlers (like `bot.onMessage`) can throw exceptions, but Switchboard will catch them and log errors. Prefer using Result types for explicit error handling.
</Warning>

## Error handling in event handlers

Event handlers can throw exceptions - they'll be caught automatically:

```typescript
bot.onMessage(async (ctx) => {
  // This is caught and logged automatically
  throw new Error('Something went wrong');
});

// Switchboard catches and logs:
// [Switchboard] Error in message handler: Error: Something went wrong
```

But using Result is more explicit:

```typescript
bot.onMessage(async (ctx) => {
  const result = await bot.sendMessage(channelId, 'Hello');
  
  if (!result.ok) {
    // Explicit error handling
    console.error('Failed to send:', result.error);
    return;  // Clean exit
  }
  
  // Continue with result.value...
});
```

## Accessing underlying errors

All Switchboard errors include a `cause` property with the original error:

```typescript
const result = await bot.sendMessage(channelId, text);

if (!result.ok && result.error instanceof MessageSendError) {
  console.error('Switchboard error:', result.error.message);
  console.error('Original error:', result.error.cause);
  //                                      ^? unknown (platform SDK error)
}
```

This is useful for debugging platform-specific issues:

```typescript
if (result.error.cause instanceof DiscordAPIError) {
  console.log('Discord error code:', result.error.cause.code);
}
```
